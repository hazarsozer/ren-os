## FINDING F1 (release-relevant): skeleton identity/overview content leaks via the "Possibly relevant now" extras section

**Severity: should-fix before release.** It directly undermines the guarantee `read_identity`/`read_overview`'s skeleton-suppression was built for (per the module docstring: *"a friend who skipped onboarding should not have placeholder boilerplate injected as if it were real identity signal"*) ‚Äî the guarantee holds for the two *dedicated* sections, but not for the wiki as a whole.

**Root cause.** `compose_wake_up_context` correctly omits the `## Who am I working with` / `## What is this project` sections while `identity.md` / `overview.md` are still skeletons (`read_identity`/`read_overview` return `""`, so nothing is added to `surfaced_pages`). But *because* nothing was added to `surfaced_pages`, these exact same files remain eligible candidates for the separate, generic "Possibly relevant now" extras path:

- `_discover_extra_candidates` (`hooks/wake-up/wakeup/__init__.py:645`) does an unfiltered `wiki_root.rglob("*.md")`, excluding only `exclude` (= `surfaced_pages`), quarantined, foreign-stamped, and archived pages. It has no knowledge of the identity/overview skeleton-suppression rule.
- `rank_extras` ‚Üí `skills.recall.lib.rank` scores every candidate; with an unfamiliar/low-overlap query (real case: a fresh project with no git history, or any query that doesn't happen to match the skeleton boilerplate) every candidate ties at the flat `RECENCY_BONUS` (0.5) since they were all written moments ago, and the tie-break (mtime, descending) ‚Äî not "is this real content" ‚Äî decides the order.
- The section's own injection code (`compose_wake_up_context`, lines ~897-903) does `text = _read_text_safe(wiki_root / rel)` ‚Äî a raw file read ‚Äî **not** `read_identity`/`read_overview`, so none of the skeleton-filtering or placeholder-stripping logic applies.

**Live reproduction (this drill, Step 2 payload, `## Possibly relevant now`):**

```
#### projects/demo/overview.md
---
title: "Project Overview"
...
---

# Project overview

<!-- What this project is, its current stage/thesis, and 3‚Äì5 load-bearing
     facts. Target ‚â§600 tokens ‚Äî a thesis, not a novel. Maintained by
     /ren:wrap; edit freely, your edits are preserved (user trust wins). -->


#### LICENSES.md
...

#### identity.md
---
title: "friend's Identity"
...
skipped_questions: []
...
---

# About friend

_This file is filled in by `/ren:interview` during onboarding. ...

## Background & current role

_One paragraph: who you are, what you focus on, what brought you here._
...
```

Both the identity page's placeholder body **and** the overview page's HTML-comment skeleton land in the payload verbatim ‚Äî the exact strings the 0.5.5 "guaranteed wake-up sections stop injecting placeholder body" fix (commit `4b3b67e`) was written to keep out.

**Persistence across sessions.** This isn't a one-shot artifact of the very first wake-up. Re-running the drill's Step 3 wake-up (after identity was filled, but *before* overview was filled) shows `projects/demo/overview.md`'s skeleton **still** leaking via extras ‚Äî it only stops once `maintain_overview` actually writes real content in Step 4 (at which point `read_overview` starts returning non-empty text, `overview.md` is added to `surfaced_pages`, and it's excluded from the extras candidate pool as a side effect ‚Äî not because of any skeleton-awareness in the extras path itself). So the leak persists for the entire pre-fill period of a friend's wiki (potentially many sessions), not just the very first one.

**Why the unit suite didn't catch this.** `tests/hooks/test_wakeup.py::test_skeleton_identity_omitted` and `::test_skeleton_overview_omitted` both assert directly against `wakeup.read_identity`/`wakeup.read_overview` in isolation ‚Äî they never call `compose_wake_up_context` on a wiki with enough sibling `.md` files for `rank_extras` to have real candidates to select from. The `project`/`wiki` pytest fixtures used elsewhere in that file create sparse, single-purpose directories (e.g. `test_absent_sources_omit_headers` writes only `l1/session-001.md`, no identity.md/overview.md/index.md/log.md siblings), so the extras path never had a skeleton page available to leak in any existing test. A `stamp_skeleton`-driven full wiki (this drill's Step 1, which mirrors the real `/ren:install` + `/ren:bootstrap-project` path) is what surfaces it.

**Suggested fix direction (not applied ‚Äî this is a drill, not a fix session):** `_discover_extra_candidates` should apply the same skeleton-emptiness check `read_identity`/`read_overview` use (or exclude `IDENTITY_FILENAME` and every project's `OVERVIEW_FILENAME` from the extras candidate pool outright whenever they're still skeleton, mirroring how they're already excluded once genuinely filled).
