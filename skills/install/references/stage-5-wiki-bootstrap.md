# Stage 5 — Master wiki skeleton bootstrap

Per ADR-015 Stage 5 + team-lead pushback **P2 (additive-only diff with explicit approve)** + ADR-017 (skeleton, not content) + ADR-027 (additive-MINOR migration pattern).

## The contract

The friend's wiki is sacred (per ADR-017). Stage 5 NEVER overwrites an existing file. Only additive writes — new files that the current wiki skeleton offers but the friend's wiki doesn't have yet.

## Procedure

### 5.1 Resolve wiki root

Target: `~/.startup-framework/wiki/` (per the path lock).

### 5.2 Delegate to the shared template loader

The template-loader procedure lives at `skills/sf-bootstrap-project/references/template-loader.md`. Single source of truth — DO NOT reimplement here.

Call the loader with:

- **Template root**: `wiki-skeleton/templates/`
- **Manifest**: `wiki-skeleton/manifest.yaml` profile `master`
- **Target root**: `~/.startup-framework/wiki/`
- **Placeholder bindings**:
  - `{{handle}}` ← `state.stage_artifacts.4.handle_written` (Stage 4 runs before Stage 5; the handle is set in the interview and lives in identity.md frontmatter)
  - `{{name}}` ← friend's display name from `identity.md` (if present); else "friend" with a warning
  - `{{today}}` ← ISO YYYY-MM-DD
  - `{{framework_version}}` ← `state.framework_version`

### 5.3 Branch on target state

The loader auto-detects:

- **Target doesn't exist** → first-run install path. Loader writes the full skeleton. No friend approval needed (no existing content to protect).
- **Target exists** → additive-diff mode. Loader computes the planned-vs-present diff; prompts the friend; writes only approved additions; existing files untouched.

The additive-diff prompt is the loader's responsibility (see `template-loader.md` § "Additive-diff mode"). Stage 5's role is to invoke the loader and forward the friend's choice.

### 5.4 Capture outcomes

After the loader completes:

```json
{
  "stage_artifacts": {
    "5": {
      "wiki_root": "~/.startup-framework/wiki",
      "skeleton_schema_version": 1,
      "additive_changes_applied": ["<path>", ...],
      "additive_changes_declined": ["<path>", ...]
    }
  }
}
```

`additive_changes_applied` is empty on a clean re-run with no new templates.
`additive_changes_declined` records any additions the friend rejected; Stage 6 doctor can re-surface them as info.

### 5.5 Friend-facing summary

For first-run install:

```
Stage 5 — Wiki skeleton at ~/.startup-framework/wiki/:
  wrote   index.md
  wrote   log.md
  wrote   identity.md           (filled by Stage 4 already)
  wrote   LICENSES.md           (to be regenerated by Stage 6)
  created research/
  created decisions/
  created alternatives/
  created patterns/
  created projects/
```

For additive-diff (existing wiki):

```
Stage 5 — Wiki at ~/.startup-framework/wiki/ already exists.
  Additive changes:
    would_write   patterns/.gitkeep          (new in framework v<n>)
    already_present  index.md
    already_present  log.md
    already_present  identity.md
    ...
  Apply additive changes? [y/N]: ...
  → applied 1 addition, 0 declined.
```

## What this stage deliberately does NOT do

- **Never overwrites an existing file.** No flag, no force mode, no implicit upgrade. (Team-lead P2 / load-bearing.)
- Doesn't rewrite `identity.md`. Stage 4 owns the identity write; Stage 5 only acknowledges its presence.
- Doesn't seed the wiki with framework-developer content. Per ADR-017.

## Edge cases

- **identity.md exists but is malformed YAML** — Stage 5 doesn't try to repair it; the loader leaves it untouched. Stage 6 doctor surfaces the issue.
- **A subdirectory exists but as a regular file with the same name** — refuse the run, surface the path conflict; suggest the friend rename the offending file.
- **The friend's wiki was created by a NEWER framework version** — `skeleton_schema_version` in their existing files exceeds the current; the loader refuses additive writes and surfaces a migration hint pointing at `/sf:update`.
- **Loader errored mid-run** — partial writes stay in place per the no-rollback policy; checkpoint records `additive_changes_applied` up to the failure point; re-run resumes.
